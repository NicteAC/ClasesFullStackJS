Rendimiento y UX

Importante: primero corregir lo que está roto, después mejorar lo que funciona. No agregar features nuevas antes de que las existentes estén estables.

Depuración — ¿Qué buscar?

Errores de lógica

Funciones que no hacen lo esperado. Condiciones invertidas. Datos que no se guardan. Componentes que no reaccionan.

Errores de UI

Elementos que no se ven bien. Layout roto en móvil. Botones que no hacen nada. Textos cortados o sin formato.

Errores de integración

API que no responde. Vuex que no actualiza. Rutas que no cargan. Firebase que rechaza operaciones.

Errores de consola

Warnings de Vue, errores de JavaScript, props sin validar, componentes no registrados.

Checklist de revisión

  • Todas las rutas cargan correctamente
  • Los formularios validan y envían datos
  • El estado de Vuex se actualiza y persiste si corresponde
  • Las llamadas a API funcionan (loading, error, datos)
  • La navegación entre páginas es fluida
  • No hay errores ni warnings en la consola del navegador
  • La UI se ve bien en distintos tamaños de pantalla
  • Los componentes reciben las props correctas
  • Los eventos ($emit) se propagan correctamente
ACTIVIDAD 1 — 40 minutos

Auditoría de tu producto

Instrucciones

1. Abre tu proyecto y recorre cada funcionalidad como si fueras un usuario.
2. Usa el checklist de la slide anterior y marca lo que pasa y lo que falla.
3. Abre la consola del navegador (F12) y anota todos los errores y warnings.
4. Crea una lista de bugs ordenada por prioridad: críticos primero, cosméticos al final.

Entregable

Un documento (puede ser un .md, .txt o papel) con:

✅ Lo que funciona correctamente.
❌ Lo que tiene errores (con descripción del bug).
⚠️ Lo que se puede mejorar.
🔢 Lista priorizada de correcciones.

Listo para comenzar

Optimización — ¿Qué mejorar?

Código

Eliminar console.log, código comentado y funciones no usadas. Nombres claros en variables y métodos.

Componentes

Extraer lógica repetida a componentes reutilizables. Props con validación. Eventos bien nombrados.

UX

Feedback al usuario: loading, mensajes de éxito/error, estados vacíos ("No hay datos aún").

Estructura

Carpetas organizadas. Store limpio. Rutas con nombres descriptivos. README actualizado.

ACTIVIDAD 2 — 60 minutos

Depuración y corrección de bugs

Instrucciones

1. Toma la lista priorizada de la actividad anterior.
2. Empieza por los bugs críticos: lo que está roto y no funciona.
3. Después corrige los bugs menores: warnings, estilos rotos, textos.
4. Si te sobra tiempo, avanza con las mejoras de código: limpiar console.log, mejorar nombres, extraer componentes.

Criterio de éxito

Al terminar deberías tener:

✅ Cero errores en la consola del navegador.
✅ Todas las funcionalidades principales operativas.
✅ El producto se puede usar de inicio a fin sin problemas.
✅ El código no tiene console.log ni código comentado.

ACTIVIDAD 3 — 20 minutos

Documentación mínima

Crea o actualiza el README.md

1. Nombre del proyecto y descripción breve.
2. Problemática que resuelve.
3. Tecnologías usadas: Vue 2, Vuex, Axios, Firebase, etc.
4. Cómo instalar: npm install, npm run serve.
5. Estructura de carpetas principal.
6. Funcionalidades implementadas.

¿Por qué importa?

El README es lo primero que ve alguien que visita tu repositorio. Si no tiene README, parece un proyecto abandonado.

Para tu portafolio, un buen README hace la diferencia entre "un proyecto más" y "un proyecto profesional".

Tip: Si tu proyecto tiene deploy, incluye el link en el README.
