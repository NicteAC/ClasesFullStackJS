Librerías UI para Vue

BootstrapVue

Bootstrap 4 adaptado a Vue. Familiar, documentado, ideal para proyectos rápidos.

Basado en Bootstrap 4, evolución lenta.

Vuetify

Material Design de Google. Componentes completos, sistema de temas, muy popular.

Bundle más pesado.

Buefy

Basada en Bulma. Ligera, minimalista, fácil de integrar. Ideal para prototipos y MVPs.

Menos componentes que Vuetify.

Element UI

Enfoque empresarial. Tablas complejas, formularios avanzados, estética corporativa.

Bundle pesado, estética menos moderna.

¿Qué es Server-Side Rendering (SSR)?

El contenido se genera en el servidor y llega como HTML listo al navegador.

Sin SSR (SPA)

El navegador recibe un HTML vacío. Vue lo llena con JavaScript.

Pantalla en blanco hasta que JS carga. Los buscadores no indexan bien.

Con SSR

El servidor envía el HTML ya renderizado. El usuario ve contenido al instante.

Mejor SEO, carga más rápida, mejor en redes lentas.

¿Cuándo usar SSR?

✅

Sí, usa SSR

Blogs, e-commerce, landing pages, portales de contenido. Todo lo que necesita SEO.

❌

No necesitas SSR

Paneles admin, dashboards internos, apps detrás de login. No necesitan SEO.

⚡

Alternativa: SSG

Generación estática: genera HTML en build time. Ideal para blogs y docs. Nuxt lo soporta.

Nuxt.js — El framework SSR para Vue

Framework que extiende Vue con SSR, rutas automáticas, layouts y generación estática.

📁

Rutas automáticas

Cada archivo en pages/ se convierte en una ruta. Sin configurar router manualmente.

🖼️

Layouts

Plantillas base en layouts/. Un layout para la app pública, otro para el admin.

🌐

SSR + SSG

Renderizado en servidor o generación estática. Se elige al crear el proyecto.

🔌

Módulos

Ecosistema de módulos: Axios, Auth, PWA, BootstrapVue, etc. Se agregan en nuxt.config.js.

Estructura de un proyecto Nuxt

mi-proyecto-nuxt/
├── pages/                ← Cada archivo = una ruta
│   ├── index.vue         ← ruta /
│   └── about.vue         ← ruta /about
├── components/           ← Componentes reutilizables
├── layouts/              ← Plantillas base
│   └── default.vue
├── store/                ← Vuex (activado automáticamente)
├── static/               ← Archivos estáticos
├── middleware/            ← Lógica antes de renderizar
├── plugins/              ← Plugins de terceros
└── nuxt.config.js        ← Configuración central

Quasar Framework

Una base de código, múltiples plataformas: web, móvil (Cordova/Capacitor) y escritorio (Electron).

Ventajas

Librería UI completa incluida. CLI propia. Soporte SSR, PWA, móvil y escritorio. Documentación clara.

Desventajas

Ecosistema abrumador al inicio. Curva de aprendizaje alta por la cantidad de opciones. El proyecto crece rápido.

# Instalar CLI y crear proyecto
npm install -g @quasar/cli
quasar create mi-proyecto

# Compilar para distintas plataformas
quasar dev                       ← web
quasar build -m pwa              ← Progressive Web App
quasar build -m cordova          ← móvil
quasar build -m electron         ← escritorio

Criterios para elegir herramientas

🔍

¿Necesito SEO?

Sí → Nuxt.js o Gridsome con SSR/SSG. No → SPA con Vue CLI funciona perfecto.

📱

¿Qué plataformas?

Solo web → Nuxt o Vue CLI. Web + móvil + escritorio → Quasar.

🎨

¿Qué estética?

Material → Vuetify. Clásica → BootstrapVue. Minimalista → Buefy. Corporativa → Element UI.

👥

¿Qué sabe el equipo?

Si conocen Bootstrap → BootstrapVue. Si es proyecto nuevo → Vuetify o Buefy.

Mejores prácticas

No mezclar librerías UI

Una sola librería por proyecto. Mezclar causa conflictos de estilos y bundles enormes.

Importación selectiva

Importar solo los componentes que usas. No importar la librería entera si solo necesitas un botón.

Seguir la estructura

Respetar las convenciones del framework (carpetas, naming, config). No pelear contra la herramienta.

Evaluar antes de adoptar

No elegir por popularidad. Evaluar si se ajusta al proyecto, equipo y necesidades reales.

Errores comunes

Sobrecargar dependencias

Cada librería suma peso al bundle. Solo agregar lo que realmente se necesita.

Ignorar la documentación

Implementar de memoria lleva a errores. La documentación oficial es la referencia.

Personalizar en exceso

Modificar demasiado los componentes base dificulta las actualizaciones futuras.

Elegir por popularidad

Lo más popular no siempre es lo mejor para tu proyecto. Evaluar caso por caso.
