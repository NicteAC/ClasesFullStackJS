El problema: prop drilling

Cuando muchos componentes necesitan los mismos datos, pasar props de padre a hijo a nieto se vuelve inmanejable.

Sin Vuex

App pasa usuario a Header, Header a NavBar, NavBar a UserBadge...

Cada componente intermedio recibe y re-envía props que no usa.

❌ Difícil de mantener

❌ Props innecesarias

Con Vuex

El estado vive en un store central. Cualquier componente accede directo.

No importa qué tan profundo esté el componente en el árbol.

✅ Acceso directo

✅ Fuente única de verdad

¿Qué es Vuex?

Librería oficial de Vue para manejar estado global de forma centralizada y predecible.

📦

Centralizado

Un solo store con todos los datos compartidos. Una fuente única de verdad.

🔍

Predecible

Solo las mutations pueden cambiar el estado. Cada cambio es rastreable en Vue DevTools.

📏

Escalable

Modules permiten dividir el store en secciones cuando la app crece.

¿Cuándo usar Vuex?

Sí, usa Vuex

Datos de usuario autenticado compartidos en toda la app.

Carrito de compras accesible desde múltiples vistas.

Formularios multi-paso que persisten entre rutas.

Dashboards con datos de varias fuentes.

No necesitas Vuex

Estado que solo usa un componente (usar data()).

Comunicación simple padre-hijo (usar props y $emit).

Apps pequeñas con pocos componentes.

Toggle de un modal o sidebar local.

State — Los datos centrales

Es el objeto que contiene todos los datos compartidos. Se accede con this.$store.state.

Definir en el store

state: {
  usuario: null,
  productos: [],
  carrito: [],
  loading: false
}

Usar en un componente

<template>
  <p>{{ $store.state.usuario }}</p>
</template>

// O con computed
computed: {
  usuario() {
    return this.$store.state.usuario
  }
}

Mutations — Cambiar el state

La única forma de modificar el state. Siempre síncronas. Se llaman con this.$store.commit('NOMBRE').

Definir mutation

mutations: {
  INCREMENT(state) {
    state.count++
  },
  SET_USUARIO(state, usuario) {
    state.usuario = usuario
  },
  AGREGAR_AL_CARRITO(state, producto) {
    state.carrito.push(producto)
  }
}

Llamar desde componente

methods: {
  incrementar() {
    this.$store.commit('INCREMENT')
  },
  setUser(user) {
    this.$store.commit('SET_USUARIO', user)
  }
}

Getters — Datos derivados

Propiedades computadas del store. Se acceden con this.$store.getters.nombre.

Definir getter

getters: {
  totalCarrito(state) {
    return state.carrito.length
  },
  carritoTotal(state) {
    return state.carrito.reduce(
      function(sum, item) {
        return sum + item.precio
      }, 0
    )
  },
  isAuthenticated(state) {
    return !!state.usuario
  }
}

Usar en componente

computed: {
  total() {
    return this.$store.getters.totalCarrito
  },
  logueado() {
    return this.$store.getters.isAuthenticated
  }
}

Actions — Lógica asíncrona

Para Firebase Auth, Firestore u operaciones async. Se llaman con this.$store.dispatch('nombre').

import { auth } from '../firebase'
import { signInWithEmailAndPassword } from 'firebase/auth'

actions: {
  login(context, credenciales) {
    context.commit('SET_LOADING', true)
    signInWithEmailAndPassword(auth, credenciales.email, credenciales.password)
      .then(function(result) {
        context.commit('SET_USUARIO', {
          uid: result.user.uid, email: result.user.email
        })
      })
      .catch(function(err) {
        context.commit('SET_ERROR', err.message)
      })
      .finally(function() {
        context.commit('SET_LOADING', false)
      })
  }
}

Caso práctico: Formulario multi-paso

Los datos del formulario se guardan en el store y persisten al navegar entre pasos.

state: {
  formulario: { nombre: '', email: '', edad: '' }
},
mutations: {
  ACTUALIZAR_CAMPO(state, payload) {
    state.formulario[payload.campo] = payload.valor
  },
  RESET_FORMULARIO(state) {
    state.formulario = { nombre: '', email: '', edad: '' }
  }
}

// En el componente
<input
  :value="$store.state.formulario.nombre"
  @input="$store.commit('ACTUALIZAR_CAMPO', { campo: 'nombre', valor: $event.target.value })"
/>

Auth con Vuex + Firebase

El usuario autenticado vive en el store. Firebase Auth maneja las credenciales, Vuex centraliza el estado.

import { auth } from '../../firebase'
import { signInWithEmailAndPassword, signOut, onAuthStateChanged } from 'firebase/auth'

state: { usuario: null, loading: false, error: null },
mutations: {
  SET_USUARIO(state, user) { state.usuario = user },
  SET_LOADING(state, val) { state.loading = val },
  SET_ERROR(state, err) { state.error = err }
},
getters: {
  isAuthenticated(state) { return !!state.usuario },
  currentUser(state) { return state.usuario }
}

Actions: login, registro, logout

actions: {
  login(context, cred) {
    context.commit('SET_LOADING', true)
    context.commit('SET_ERROR', null)
    return signInWithEmailAndPassword(auth, cred.email, cred.password)
      .then(function(result) {
        context.commit('SET_USUARIO', {
          uid: result.user.uid, email: result.user.email
        })
      })
      .catch(function(err) { context.commit('SET_ERROR', err.message) })
      .finally(function() { context.commit('SET_LOADING', false) })
  },
  logout(context) {
    return signOut(auth).then(function() {
      context.commit('SET_USUARIO', null)
    })
  },
  escucharAuth(context) {
    onAuthStateChanged(auth, function(user) {
      if (user) {
        context.commit('SET_USUARIO', { uid: user.uid, email: user.email })
      } else {
        context.commit('SET_USUARIO', null)
      }
    })
  }
}

Modules — Dividir el store

Cada módulo tiene su propio state, mutations, actions y getters. Con namespaced: true se accede con prefijo.

// store/modules/auth.js
export default {
  namespaced: true,
  state: { usuario: null, loading: false, error: null },
  mutations: { /* SET_USUARIO, SET_LOADING, SET_ERROR */ },
  actions: { /* login, logout, escucharAuth */ },
  getters: { /* isAuthenticated, currentUser */ }
}

// store/index.js
import auth from './modules/auth'
export default new Vuex.Store({ modules: { auth } })

// En componente
this.$store.dispatch('auth/login', datos)
this.$store.getters['auth/isAuthenticated']
this.$store.state.auth.usuario

Helpers: mapState, mapGetters, mapActions

Atajos para no escribir this.$store en cada computed o method.

import { mapState, mapGetters, mapActions } from 'vuex'

export default {
  computed: {
    ...mapState('auth', ['usuario', 'loading', 'error']),
    ...mapGetters('auth', ['isAuthenticated', 'currentUser'])
  },
  methods: {
    ...mapActions('auth', ['login', 'logout', 'escucharAuth'])
  }
}

// Ahora se usan como propiedades/métodos locales:
// this.usuario, this.isAuthenticated, this.login({...})

Errores comunes

Vuex para todo

No metas todo en el store. data() sigue siendo válido para estado local.

Mutar sin mutation

Nunca hagas this.$store.state.count++. Siempre usa commit.

Async en mutations

Las mutations son síncronas. Firebase Auth va en actions.

Store gigante

Divide en modules. Un archivo enorme es difícil de mantener.

Instrucciones

  1. Instala Vuex y crea un store con un módulo auth que use Firebase Auth (login, logout, escucharAuth).
  2. Agrega al menos otra sección de datos (carrito, tareas, formulario) con sus mutations.
  3. Implementa getters para datos derivados (isAuthenticated, total items, etc.).
  4. Usa los datos del store en al menos dos componentes distintos sin props.
  5. Inicia el listener escucharAuth en App.vue para mantener la sesión al recargar.
  6. Bonus: usa mapState/mapGetters/mapActions en los componentes.
