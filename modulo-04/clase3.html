Bloque 1 · Asincronía

JavaScript es Single-Thread

🧵 Un solo hilo

JavaScript ejecuta una sola tarea a la vez. No puede hacer dos cosas simultáneamente de forma nativa.

Esto significa que si una tarea tarda mucho (leer un archivo, llamar a una API), bloquearía todo el código siguiente.

🔄 Solución: asincronía

En lugar de esperar, JavaScript delega las tareas lentas al entorno (navegador o Node.js) y sigue ejecutando el resto del código.

Cuando la tarea lenta termina, la respuesta vuelve a la cola de JavaScript para procesarse.

Analogía: el mozo anota tu pedido, lo manda a la cocina y sigue atendiendo otras mesas. No espera parado que la comida esté lista.
Bloque 1 · Call Stack y Event Loop

Call Stack y Event Loop

📋

Call Stack

Pila de llamadas. Registra qué función se está ejecutando. LIFO: última en entrar, primera en salir.

🌐

Web APIs

El navegador provee: setTimeout, fetch, eventos del DOM. Corren fuera del hilo de JS.

📬

Callback Queue

Cola donde esperan los callbacks listos para ejecutarse cuando el Call Stack esté vacío.

🔄

Event Loop

Vigila el Call Stack. Cuando está vacío, mueve el siguiente callback de la Cola al Stack.

console.log("1 - Inicio");
setTimeout(() => console.log("3 - Timeout (0ms)"), 0);
console.log("2 - Fin");
// Imprime: 1 → 2 → 3 (el timeout va a la cola aunque sea 0ms)
Bloque 1 · Timers

setTimeout y setInterval

// setTimeout: ejecutar UNA VEZ después de X milisegundos
const id1 = setTimeout(() => {
  console.log("Ejecutado después de 2 segundos");
}, 2000);

// clearTimeout: cancelar antes de que se ejecute
clearTimeout(id1); // cancela el timeout de arriba

// setInterval: ejecutar REPETIDAMENTE cada X milisegundos
let cuenta = 0;
const id2 = setInterval(() => {
  cuenta++;
  console.log(`Segundo: ${cuenta}`);
  if (cuenta === 5) clearInterval(id2); // detener a los 5
}, 1000);
Siempre guardar el ID retornado para poder cancelar el timer cuando ya no sea necesario y evitar memory leaks.
Bloque 1 · Callbacks

Callbacks y Callback Hell

// Callback: función que se pasa como argumento para llamar después
function obtenerUsuario(id, callback) {
  setTimeout(() => {
    callback({ id: id, nombre: "Ana" });
  }, 1000);
}
obtenerUsuario(1, usuario => console.log(usuario));

// ❌ CALLBACK HELL: anidamiento que dificulta la lectura
obtenerUsuario(1, u => {
  obtenerPedidos(u.id, pedidos => {
    procesarPago(pedidos[0], pago => {
      console.log("Procesado"); // 😵 difícil de mantener
    });
  });
});
El Callback Hell es la razón por la que nacieron las Promesas y luego async/await.
Bloque 2 · Promesas

Promesas — Promises

// Una promesa representa un valor futuro: éxito o fracaso
const promesa = new Promise((resolve, reject) => {
  const exito = true;
  if (exito) resolve("✅ Datos obtenidos");
  else reject("❌ Error al obtener datos");
});

// Consumir con .then() / .catch() / .finally()
promesa
  .then(datos => console.log("Éxito:", datos))
  .catch(error => console.error("Error:", error))
  .finally(() => console.log("Siempre se ejecuta"));
Una promesa tiene 3 estados: pending (esperando) · fulfilled (resuelta) · rejected (rechazada)
Bloque 2 · async/await

async / await — La Forma Moderna

🔗 Con .then() (prometido)

obtenerUsuario(1)
  .then(u => obtenerPedidos(u.id))
  .then(p => procesarPago(p[0]))
  .then(r => console.log(r))
  .catch(e => console.error(e));

✅ Con async/await (mejor)

async function flujo() {
  try {
    const u = await obtenerUsuario(1);
    const p = await obtenerPedidos(u.id);
    const r = await procesarPago(p[0]);
    console.log(r);
  } catch(e) {
    console.error(e);
  }
}
Bloque 2 · Promesas avanzadas

Promesas en Paralelo

// Promise.all: espera TODAS · falla si una falla
const [users, posts] = await Promise.all([
  fetch("https://jsonplaceholder.typicode.com/users").then(r => r.json()),
  fetch("https://jsonplaceholder.typicode.com/posts").then(r => r.json())
]);

// Promise.allSettled: espera TODAS · nunca falla · informa cada resultado
const resultados = await Promise.allSettled([promesa1, promesa2]);
resultados.forEach(r => {
  if (r.status === "fulfilled") console.log(r.value);
  else console.error(r.reason);
});

// Promise.race: resuelve con la PRIMERA que termine
const rapida = await Promise.race([promesaA, promesaB]);
Bloque 2 · Errores

Manejo de Errores Asíncronos

async function obtenerDatos(url) {
  try {
    const respuesta = await fetch(url);

    // fetch NO lanza error en 404/500 — hay que verificar ok
    if (!respuesta.ok) {
      throw new Error(`HTTP Error: ${respuesta.status}`);
    }

    const datos = await respuesta.json();
    return datos;
  } catch (error) {
    // Errores de red Y errores lanzados manualmente
    console.error("Error:", error.message);
    throw error; // re-lanzar para que el llamador lo maneje
  } finally {
    console.log("Finalizó la petición");
  }
}
⚠️ fetch() solo rechaza la promesa ante errores de red. Errores HTTP (404, 500) hay que verificarlos con response.ok.

Bloque 3 · API REST

¿Qué es una API REST?

📡 Definición

Application Programming Interface + REpresentational State Transfer.

Forma estándar de comunicación entre sistemas a través de HTTP. El cliente pide, el servidor responde con datos en JSON.

🌍 Ejemplos del mundo real

  • Twitter → publicar tweets, obtener timeline
  • Spotify → buscar canciones, crear playlists
  • Google Maps → coordenadas, rutas
  • GitHub → repositorios, commits, usuarios
  • Stripe → pagos, tarjetas, suscripciones
Nuestra app: cliente (navegador con JS) → HTTP Request → servidor → JSON Response → mostrar en pantalla

Bloque 3 · Status Codes

Códigos de Estado HTTP

✅ 2xx — Éxito

  • 200 OK — solicitud exitosa
  • 201 Created — recurso creado
  • 204 No Content — éxito sin cuerpo

⚠️ 4xx — Error cliente

  • 400 Bad Request — datos inválidos
  • 401 Unauthorized — sin autenticación
  • 403 Forbidden — sin permiso
  • 404 Not Found — no existe

❌ 5xx — Error servidor

  • 500 Internal Server Error
  • 502 Bad Gateway
  • 503 Service Unavailable
En JavaScript: response.status → el código · response.ok → true si el código está entre 200 y 299

Bloque 4 · Fetch POST

Fetch POST — Crear Recursos

async function crearPost(titulo, cuerpo) {
  try {
    const respuesta = await fetch("https://jsonplaceholder.typicode.com/posts", {
      method: "POST",
      headers: { "Content-Type": "application/json" },
      body: JSON.stringify({ title: titulo, body: cuerpo, userId: 1 })
    });

    if (!respuesta.ok) throw new Error(`Error ${respuesta.status}`);

    const nuevo = await respuesta.json();
    console.log("✅ Creado con id:", nuevo.id, "|", nuevo.title);
  } catch (e) { console.error(e.message); }
}
crearPost("Mi primer post", "Contenido del post");

Fetch PUT y DELETE

PUT Actualizar

async function actualizarPost(id, data) {

    method: "PUT",
    headers: { "Content-Type": "application/json" },
    body: JSON.stringify(data)

async function eliminarPost(id) {
  const r = await fetch(`/posts/${id}`, {
    method: "DELETE"

Headers HTTP

const opciones = {
  method: "POST",

console.log(r.status, r.statusText);

Bloque 4 · Patrón completo

El Patrón Completo de Fetch

// Función genérica reutilizable para cualquier método
async function solicitud(url, metodo = "GET", datos = null) {
  const opciones = {
    method: metodo,
    headers: { "Content-Type": "application/json" }
  };
  if (datos) opciones.body = JSON.stringify(datos);

  const r = await fetch(url, opciones);
  if (!r.ok) throw new Error(`HTTP ${r.status}: ${r.statusText}`);
  return r.status === 204 ? null : r.json();
}

// Usar la función genérica
const BASE = "https://jsonplaceholder.typicode.com";
const usuarios = await solicitud(`${BASE}/users`);
const nuevo = await solicitud(`${BASE}/posts`, "POST", { title: "Test" });
await solicitud(`${BASE}/posts/1`, "DELETE");
